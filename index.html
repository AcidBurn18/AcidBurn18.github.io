Architecture

System view.

Home Lab

Reproducible local infrastructure

Proxmox, Docker, and documented service placement keep the lab understandable after every change.

  • Goals: stable services, readable failure domains, repeatable rebuilds.
  • Design: split edge, workloads, and utility services into explicit layers.
  • Flow: ingress and DNS are treated as first-class infrastructure.
Networking

Traffic planes with clear boundaries

The network story separates browsing, VPN, reverse proxy, and local app traffic so troubleshooting is fast.

  • Goals: keep remote access explicit and avoid hidden routing paths.
  • Design: OPNsense, AdGuard, Unbound, Traefik, Twingate, and Tailscale work as distinct layers.
  • Flow: every path can be drawn, traced, and explained.
Monitoring

Visibility before guesswork

Observability turns the lab into a platform: logs, dashboards, and a visible change path.

  • Goals: detect issues early and keep post-change diagnosis simple.
  • Design: Promtail, Loki, Grafana, and host-level logging sit alongside services.
  • Flow: app issues should lead to one obvious place to inspect first.
Automation

Documentation-first operations

Infrastructure and writeups move together so the portfolio remains honest as the lab evolves.

  • Goals: reduce tribal knowledge and make the next change easier than the last.
  • Design: IaC, runbooks, and public writing reinforce the same operating model.
  • Flow: every meaningful change should leave a paper trail.

Engineering Principles

Short rules that keep the work sane.

Reproducible

Infrastructure should be rebuildable.

If a change cannot be explained or repeated, it is not ready to be trusted.

Automated

Automation should remove repetitive work.

Manual steps belong in runbooks only when the task is genuinely rare.

Observable

Observability is mandatory.

If a service matters, there should be a clear way to see when it is unhealthy.

Secure

Security should be designed in.

Ingress, remote access, and credentials are handled as architecture decisions, not afterthoughts.

Documented

Documentation is part of the product.

A system is easier to maintain when the reasoning lives with the implementation.

Simple

Prefer simplicity over unnecessary complexity.

Each extra moving piece must earn its place by making the system easier to operate.
